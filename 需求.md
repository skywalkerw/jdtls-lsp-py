reverse-design 的执行流程分步骤：

- **step1**：分析工程概要信息  
- **step2**：列出 REST 接口清单  
- **step3**：列出数据库表清单  
- **step4**：针对每一个 REST 接口向下查找调用链  
- **step5**：针对每一个数据库表向上查找调用链  
- **step6**：识别调用链中的关键业务位置  
- **step7**：针对关键业务，补全实现细节  
- **step8**：汇总输出分析结果，总分结构，从宏观到微观看整个项目  

---

## 与 `jdtls-lsp reverse-design` 的对应关系（便于对照代码与 CLI）

| step | 含义 | 工具侧 |
| --- | --- | --- |
| step1 | 工程概要 | `reverse-design scan`（模块）、`symbols`（按包顶层类型）；`bundle` 内默认生成 `modules.json`、`symbols-by-package.json` |
| step2 | REST 清单（**静态入口扫描**，与 `entrypoints` 并列） | `reverse-design rest-map` → `data/rest-map.json` |
| step3 | 数据库表清单 | `reverse-design db-tables` / `bundle` 内 `tables-manifest.json` |
| step4 | 每接口向下调用链 | `bundle --rest-callchains-down` → `data/callchain-down-rest-*.md`（文末 JSON） |
| step5 | 每表向上调用链 | `bundle --table-callchains-up` → `data/callchain-up-table-*.md`；`--queries` 为**关键字向上**（同属向上族，非按表） |
| step6 | 链上关键业务 | 向下链 JSON 内 `businessCandidate` / `keyMethods`；`--business-summary` → `business.md` |
| step7 | 补全实现细节 | **未在 bundle 内全自动**：用 `jdtls-lsp analyze`、`callchain-up`/`down` 单点或 IDE 深挖 |
| step8 | 汇总输出 | `bundle` 收尾写入 `index.md` + stdout 摘要 JSON；与 `business.md`、各 `*-summary.json` 共同构成「宏观→微观」阅读入口 |

**说明**：`bundle` 内**技术执行顺序**为 A1→A2→A2.5→A3→（可选）`--queries`→`--table-callchains-up`→`--rest-callchains-down`→`--business-summary`→`index.md`，与上表 **step 编号叙述顺序** 不完全一致；语义上仍分别对应 step1–step6 与 step8。
